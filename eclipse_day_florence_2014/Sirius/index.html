Domain Specific Language

Improve the communication

So as we saw on the previous slides, one domain means one dedicated language and specific representations. The purpose of domain specific language is mainly to improve the communication. But giving a common language is not enough, what people needs is...

DSL based tools

  • Are easy to take in hand
  • Guide the user to the right way
...tools adapted to their own DSL. These tools are easier to take in hand as the users already know the vocabulary. So they have more chanced to be well received by the end user. Even better, these kind of dedicated tools guide the users thanks to workflows and help them to produce valid models and so valid systems.

A bit of history

Height years ago, Thales tried to work with UML and try to adapt the notation by developing their own UML profiles.

Thales previous experiences with UML profiling

  • Poor adoption
  • Metamodels constrained by UML concepts
  • Representations constrained by existing UML diagrams
The result of this was a poor adoption by their users. Because, as they were mainly system engineers, they don't know anything about UML. So for them it was another new language to learn. And with it, they couldn't represents all the concepts they needs. Even worth for them, the resulting representations were constrained by the existing UML diagrams which do not fit their expectations.

Needs for DSLs

  • More freedom in representations
  • Heavier and more technical developments
  • Originally 2 or 3 foreseen modeling tools
So they conclude that they need DSLs. They were aware that using DSL means developing new adapted tools which results in heavier and more technical developments. They came to see Obeo and asked us for a tool to get more "freedom" in the representations. At this moment, they imagined to develop 2 or 3 different workbenches.

Sirius in a nutshell

  • A tool to quickly define DSL based custom multi-view workbenches with dedicated representations
  • Based on Eclipse Modeling Technologies : no need to be an expert for most uses
  • New Eclipse Project : but already proven technology

The result of this collaboration is "Sirius". In a nutshell, Sirius allows you to create very quickly multi-view graphical workbenches. You can create modelers based on shapes, trees and tables. It works with any EMF compatible data model. Sirius is based on many modeling technologies like EMF and GMF Runtime but you do not need to be an expert of these technologies to use it and get a working modeler. Indeed, one of the purpose of Sirius is to bring all the cool stuff existing in the Eclipse Modeling technologies to the masses and package them in an easy to use way.

Sirius goes open source

Released in 2013 as an official Eclipse project

After several years of collaboration, Sirius is now open source. And the first release of Sirius as an official Eclipse project will occured next june with the Luna release train. So, Sirius is both a new Eclipse project, and also a proven and mature techno developped in collaboration by Thales and Obeo for many years now.

What benefits

  • For developers : reduce cost & complexity
    • No need to be an Eclipse modeling expert
    • Dynamic & iterative (live) development
  • For end-users :
    • Tooling adapted to their needs and workflows
    • Improved experience (compared to raw GMF)

What are the benefits for you if you adopt Sirius ? If you're a developper it reduces drastically the cost and complexity of creating your own modelers. You don't need to be a GMF expert to create a graphical modeler. Sirius provides also a dynamic and iterative way to develop modelers with no code generation. For the end user, he gets a tooling completely adapted to its needs and its workflows, which provides a straightforward user experience.

Two parts for two kinds of users

Specification Environment : Specifier/Developer

Runtime Environment : End-user

How does it works ? There are two different parts corresponding to two kind of users.

Specification Environment

  • Define custom multi-view workbenches : Diagrams, Tree, Tables
  • With little technical knowledge
  • Get a working modeler fast : instant feedback
  • Highly customizable : native tooling, Java or extension points

First, the specification environment, is used by the developer of the modeler to define how the final modeler will work and what it will look.

Runtime Environment

  • Work with data models: not just a drawing tool
  • Executes the specification : No code generation
  • Environment adapted to user needs (Viewpoint-based)
  • Viewpoint-based
  • Simplified workflow : Modeling Project

Using a modeling workbench based on Sirius means for the end user that : * He works with models and not just a drawing tool which means that he could use the data for other purposes. * He has a good feeling working with the modeler as it is adapted to its needs. The Sirius runtime environment is installed on the end-user machine and his role is to execute the specification you defined. It interprets the configuration file and there is no code generation or any other intermediate step, it is purely dynamic.
